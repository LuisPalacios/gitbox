## 02_Database.dbml -- Config Data Model

Model the JSON config structure as DBML tables.

**Persisted tables** (from `pkg/config/config.go`):

Config, GlobalConfig, WindowState, SSHGlobal, GCMGlobal, TokenGlobal, Account, SSHConfig, GCMConfig, Source, Repo, Mirror, MirrorRepo

**Rules:**

- Map keys as `varchar [pk]` with note: "In JSON, this is the map key, not a stored field"
- All fields from Go structs with correct types
- Optional fields marked `[note: 'optional']`
- Relationships via `Ref:` syntax

**Runtime-only types** (separate `TableGroup`, marked as transient):

RemoteRepo, RepoInfo, PushMirrorInfo, StatusState (enum), RepoStatus, SetupResult, StatusResult, DiscoveredMirror, DiscoveryResult, MirrorSummary

**API Surface** (separate `TableGroup "API Surface"`, using DBML `Note` blocks):

Document the top 20-30 exported functions that a new `pkg/` consumer would call. For each:

- Full Go signature: `func FunctionName(param Type) (Result, error)`
- Error semantics: does it return `error`? Can caller safely ignore it? What errors are expected?
- I/O type: file, network, git subprocess, or pure computation
- Calling sequence notes: "Must call A before B", "Call C after D to persist changes"
- Union-like types: where a struct field can hold different kinds, document the variants and how to distinguish them

Example:

```dbml
TableGroup "API Surface" {
  // pkg/credential
  Note credential_VerifyToken {
    '''
    func VerifyToken(provider, url, token string) error
    — I/O: network (HTTP GET to provider API)
    — Returns error if token is invalid or network fails
    — Must be called before StoreToken to validate input
    '''
  }
}
```

This group prevents implementers from guessing how to call `pkg/` functions.

## 03_API_Contract.yaml -- OpenAPI 3.0

Two sections using tags:

**Section A -- Provider APIs Consumed** (tags: github, gitlab, gitea, bitbucket):

Document every HTTP endpoint the Go code actually calls. For each: method, path, auth header, query params, request/response schemas, pagination, error codes.

**Section B -- Wails Bindings** (tag groups: wails-config, wails-status, etc.):

All public `App` methods from `cmd/gui/app.go` documented as `/wails/{MethodName}` paths with `x-wails-binding: true`. Include: parameters, return types, Wails events emitted.

**Async event schemas (critical):**

Operations that emit progress or completion events MUST document them using an `x-events` extension on the operation. For each event: name, payload schema, emission frequency.

```yaml
x-events:
  - name: "clone:progress"
    payload:
      phase: string    # "Receiving objects", "Resolving deltas"
      percent: integer
    frequency: "emitted per git output line"
  - name: "clone:done"
    payload:
      repoKey: string
      error: string    # empty on success
```

This prevents implementers from having to reverse-engineer async behavior from the GUI frontend source.

**Components/schemas:** Reusable definitions for all Go struct types.

## 04_Features/ -- Gherkin Feature Files

Seven `.feature` files matching package boundaries:

| File | Domain |
| --- | --- |
| `configuration.feature` | Config init, load, save, migrate, validate |
| `accounts.feature` | Account CRUD, rename cascading |
| `credentials.feature` | Token/GCM/SSH setup, verify, delete, resolution chain |
| `repositories.feature` | Discover, clone, pull, fetch, status, folder structure |
| `mirrors.feature` | Mirror groups, push/pull setup, status, discover |
| `identity.feature` | Per-repo identity, global identity |
| `gui.feature` | Onboarding, views, themes, progress, autostart |

**Rules:**

- Strict Gherkin: Feature, Background, Scenario/Scenario Outline, Given/When/Then
- Scenario Outline with Examples for repeated patterns
- Concrete values (real provider names, URLs, field values)
- Each scenario maps to actual code behavior (read the source)
- Each feature file MUST include at least one `@error` scenario (network failure, invalid input, permission denied) and one `@empty` scenario (no accounts, no repos, empty config). These prevent implementers from writing happy-path-only code

## 05_UI_Design.json -- Design Tokens (W3C DTCG)

```json
{
  "$name": "gitbox",
  "dark": { "...all dark theme tokens..." },
  "light": { "...all light theme tokens..." },
  "shared": { "...theme-independent tokens..." }
}
```

Each token: `{ "$value": "...", "$type": "color|dimension|shadow|fontFamily|string", "$description": "..." }`

**Extract from:**

- `App.svelte` `:global([data-theme])` blocks -- CSS custom properties
- `theme.ts` -- status colors, credential badge palettes, symbols, labels
- Hardcoded inline styles -- accent colors, font sizes, border-radius

**Token groups:**

- Theme colors (bg, border, text, brand, ring, overlay, shadow, spin)
- Status palette (12 states x 2 themes)
- Credential badges (5 states x bg/border/text x 2 themes)
- Accent colors (link, danger, warning, success)
- Font family, sizes (9px-22px), weights (400-700)
- Border-radius values
- Unicode status symbols and mirror direction arrows

## 00_\<Name\>_Prompt.md -- Implementation Prompt

Self-contained prompt for implementing a new consumer of `pkg/` (TUI, web UI, bot, etc.). Only created on initial generation, not overwritten on updates.

**Use `ai_blueprint/00_TUI_Prompt.md` as the gold-standard example.** It demonstrates every required section and pattern described below.

**Required structure:**

1. **Execution instructions** -- Tell the AI to plan first, state assumptions, and verify at each phase
2. **Context** -- What the project is, what `pkg/` provides, what already exists
3. **Target behavior** -- What the new binary/mode does, detection logic, naming
4. **Blueprint reference** -- Point to each `ai_blueprint/` deliverable and explain what to extract from it
5. **Phased implementation** -- Break into 2-4 phases, each with a clear scope and a mandatory verification gate (`go build`, `go vet`, manual test). NEVER ask for everything in one pass
6. **pkg/ usage rules** -- Explicit constraints to prevent the most common AI-generation bugs:
   - Never duplicate logic that exists in `pkg/` — search for helpers first
   - Always use typed helpers instead of string formatting
   - Always check error returns — no blank `_` identifiers for errors
   - Document calling sequences for multi-step operations
7. **Async pattern** -- Provide a concrete code example of the framework's async pattern (e.g., `tea.Cmd` for Bubble Tea, `fetch` + state for React). Don't let the AI invent its own
8. **State management rules** -- How screens/components clean up state, when to reload config, how to handle navigation back
9. **Styling** -- Reference `05_UI_Design.json` tokens, map them to the target framework's styling API
10. **Platform rules** -- Target OS, terminal compatibility, font/symbol fallbacks
11. **Testing protocol** -- Build, vet, and test commands for each phase. Include both happy-path and error-path manual tests
12. **What NOT to do** -- Explicit guardrails (don't modify `pkg/`, don't add flags, etc.)
